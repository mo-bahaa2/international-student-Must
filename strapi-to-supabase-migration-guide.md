# Strapi to Supabase Migration Guide (Frontend + Backend Integration)

## Goal

Use this guide to transform a frontend that currently talks to Strapi into one that talks to Supabase.

It covers:

- Architecture differences
- Data modeling and SQL setup
- Auth migration
- Storage/media migration
- Frontend API replacement patterns
- RLS (Row Level Security)
- Realtime, server-side operations, and deployment
- Troubleshooting and cutover checklist

---

## 1. Concept Mapping: Strapi vs Supabase

| Strapi                                 | Supabase equivalent                          |
| -------------------------------------- | -------------------------------------------- |
| Content Types                          | PostgreSQL tables                            |
| REST/GraphQL API generated by Strapi   | Supabase PostgREST API + supabase-js client  |
| Users & Permissions plugin             | Supabase Auth + RLS policies                 |
| Upload plugin/media library            | Supabase Storage buckets                     |
| Lifecycle hooks/controllers            | Postgres triggers/functions + Edge Functions |
| Role-based permissions in Strapi admin | PostgreSQL roles + RLS + app-side checks     |

Key shift:

- In Strapi, app rules are mostly in Strapi backend code.
- In Supabase, data access rules should be enforced in SQL/RLS first.

---

## 2. Migration Strategy (Recommended)

1. Inventory all Strapi entities, relations, and endpoints.
2. Create equivalent SQL schema in Supabase.
3. Implement RLS policies before connecting frontend.
4. Move media/files to Storage buckets.
5. Replace frontend API calls endpoint-by-endpoint.
6. Migrate auth flows (login/register/invite/reset).
7. Run side-by-side validation with production-like data.
8. Cut over read traffic, then write traffic.

## 3. Data Model Migration

### 3.1 Extract Strapi model inventory

For each Strapi content type, collect:

- Fields and types
- Required/optional fields
- Unique constraints
- Relations (1:1, 1:N, N:N)
- Draft/publish requirements
- Slug/SEO fields
- Localization behavior

### 3.2 Convert to PostgreSQL tables

Example conversion:

```sql
create table if not exists public.articles (
  id uuid primary key default gen_random_uuid(),
  title text not null,
  slug text not null unique,
  body text,
  is_published boolean not null default false,
  published_at timestamptz,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  constraint articles_title_not_blank check (length(trim(title)) > 0)
);
```

### 3.3 Relations

- 1:N: foreign key on child table
- N:N: join table

```sql
create table if not exists public.tags (
  id uuid primary key default gen_random_uuid(),
  name text not null unique
);

create table if not exists public.article_tags (
  article_id uuid not null references public.articles(id) on delete cascade,
  tag_id uuid not null references public.tags(id) on delete cascade,
  primary key (article_id, tag_id)
);
```

### 3.4 Updated timestamp automation

```sql
create or replace function public.set_updated_at_column()
returns trigger
language plpgsql
as $$
begin
  new.updated_at = now();
  return new;
end;
$$;

create trigger trg_articles_set_updated_at
before update on public.articles
for each row execute function public.set_updated_at_column();
```

## 4. RLS Design (Critical)

Always enable RLS on exposed schemas (especially public):

```sql
alter table public.articles enable row level security;
```

### 4.1 Public read-only content

```sql
drop policy if exists articles_select_anon on public.articles;
create policy articles_select_anon on public.articles
for select to anon
using (is_published = true);
```

### 4.2 Authenticated editor CRUD (example)

```sql
drop policy if exists articles_select_auth on public.articles;
create policy articles_select_auth on public.articles
for select to authenticated
using (true);

create policy articles_insert_auth on public.articles
for insert to authenticated
with check (true);

create policy articles_update_auth on public.articles
for update to authenticated
using (true)
with check (true);

create policy articles_delete_auth on public.articles
for delete to authenticated
using (true);
```

### 4.3 Owner-only pattern

```sql
-- requires owner_id uuid column
create policy profiles_select_owner on public.profiles
for select to authenticated
using (owner_id = auth.uid());
```

Important:

- UPDATE needs matching SELECT access.
- Missing SELECT policy often looks like "update worked but 0 rows changed".

## 5. Authentication Migration

Strapi auth is usually JWT-based via users-permissions plugin.
Supabase auth provides:

- Email/password
- Magic links
- OAuth providers
- Session management
- User metadata

### 5.1 Basic frontend auth setup

```ts
import { createClient } from "@supabase/supabase-js";

export const supabase = createClient(
  import.meta.env.VITE_SUPABASE_URL,
  import.meta.env.VITE_SUPABASE_ANON_KEY,
);
```

### 5.2 Login

```ts
await supabase.auth.signInWithPassword({ email, password });
```

### 5.3 Logout

```ts
await supabase.auth.signOut();
```

### 5.4 Profile table

Keep app profile fields in a table like public.profiles linked to auth.users(id).

## 6. Storage / Media Migration

Strapi Upload plugin -> Supabase Storage buckets.

### 6.1 Create bucket

- Example: article-images (public)

### 6.2 Upload from frontend

```ts
const { error } = await supabase.storage
  .from("article-images")
  .upload(`articles/${articleId}/${file.name}`, file, { upsert: true });
```

### 6.3 Public URL

```ts
const { data } = supabase.storage.from("article-images").getPublicUrl(path);
const imageUrl = data.publicUrl;
```

### 6.4 Storage policies

Set INSERT/SELECT/UPDATE/DELETE policies as needed. For upsert workflows, ensure required permissions are granted.

## 7. Frontend API Call Replacement Patterns

## 7.1 List content

Strapi:

- GET /api/articles?pagination[page]=1&pagination[pageSize]=10&sort=createdAt:desc

Supabase:

```ts
const from = 0;
const to = 9;

const { data, error, count } = await supabase
  .from("articles")
  .select("*", { count: "exact" })
  .order("created_at", { ascending: false })
  .range(from, to);
```

## 7.2 Filter

```ts
const { data } = await supabase
  .from("articles")
  .select("*")
  .eq("category", "news")
  .ilike("title", "%ai%");
```

## 7.3 Get one row

```ts
const { data, error } = await supabase
  .from("articles")
  .select("*")
  .eq("slug", slug)
  .maybeSingle();
```

## 7.4 Create

```ts
const { error } = await supabase.from("articles").insert({
  title,
  slug,
  body,
  is_published: false,
});
```

## 7.5 Update

```ts
const { error } = await supabase
  .from("articles")
  .update({ title, updated_at: new Date().toISOString() })
  .eq("id", id);
```

## 7.6 Delete

```ts
const { error } = await supabase.from("articles").delete().eq("id", id);
```

## 7.7 Joins / nested select

```ts
const { data } = await supabase
  .from("articles")
  .select("id,title,author:profiles(full_name),article_tags(tag:tags(name))");
```

## 8. Draft/Publish Behavior

Strapi often has Draft & Publish.
Replicate with columns like:

- is_published boolean
- published_at timestamptz

Then enforce visibility through RLS:

- anon sees only is_published = true
- authenticated editors can see all

## 9. Realtime (Optional)

If you need live updates:

```ts
const channel = supabase
  .channel("articles-feed")
  .on(
    "postgres_changes",
    { event: "*", schema: "public", table: "articles" },
    (payload) => {
      console.log(payload);
    },
  )
  .subscribe();
```

Use Realtime only where needed (chat, live dashboards, notifications).

## 10. Server-Side Logic

Use these patterns instead of Strapi custom controllers/services:

- SQL functions (for transactional DB logic)
- Triggers (auto updates, derived fields)
- Edge Functions (secure server logic, API integrations, secrets)

Use service_role key only on trusted server environments, never in browser frontend.

## 12. Data Migration Steps

1. Export Strapi data (JSON/CSV/DB dump).
2. Normalize field names and relation IDs.
3. Insert parent tables first, then child/join tables.
4. Migrate media files to Storage and update file URLs.
5. Validate row counts per entity.

Validation SQL examples:

```sql
select 'articles' as table_name, count(*) from public.articles
union all
select 'tags', count(*) from public.tags;
```

## 13. Performance Checklist

- Add indexes for frequent filters and joins.
- Use pagination with range.
- Avoid selecting all columns by default.
- Use explain analyze for heavy queries.
- Precompute expensive aggregates when needed.

Example index:

```sql
create index if not exists idx_articles_published_at
on public.articles (is_published, published_at desc);
```

---

## 14. Security Checklist

- Enable RLS for every exposed table.
- Create least-privilege policies.
- Do not expose service_role key to frontend.
- Keep privileged operations in Edge Functions or SQL functions with controlled grants.
- Validate file type/size in Storage policies.
- Audit anon policies before production.

## 15. Common Issues and Fixes

### Issue: Query returns empty data but rows exist

Possible cause:

- RLS policy missing for authenticated role.

Fix:

- Add authenticated select policy for table.

### Issue: Insert/update fails silently or returns permission error

Possible cause:

- Missing insert/update policy or missing select policy for update flow.

Fix:

- Add matching insert/update/select RLS policies.

### Issue: Auth works locally but fails after deployment

Possible cause:

- Wrong Site URL/Redirect URL in Supabase auth settings.

Fix:

- Add exact production URL(s) in Supabase Authentication URL configuration.

### Issue: Files upload but URL not visible

Possible cause:

- Bucket private or select policy missing.

Fix:

- Update bucket visibility/policies or use signed URLs.

## 17. Production Cutover Plan

1. Freeze Strapi writes (or run dual-write temporarily).
2. Final delta migration to Supabase.
3. Run verification queries.
4. Switch frontend env vars to Supabase project.
5. Deploy frontend.
6. Monitor logs/errors.
7. Keep rollback path for a short window.

- Supabase client initialized once.
- All Strapi fetch helpers replaced by Supabase service functions.

## 20. What to Keep from Strapi Era

You can still preserve your:

- Frontend component structure
- Page-level services/repositories
- Validation patterns
- UI states and UX

Only the data access layer and auth/storage integration need to change deeply.
